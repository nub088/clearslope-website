# Clear Slope Digital — SEO Redesign Design Doc

**Date:** 2026-05-03
**Author:** Mo Habib
**Status:** Approved (brainstorm complete) — pending implementation plan

## Objective

Overhaul `clearslopedigital.com` from a thin single-page Astro site into a multi-page, content-rich, schema-correct property that can rank in Google for service-business prospects. The "phonebook effect" (showing up for branded/navigational queries) is the floor, not the ceiling. The ceiling is ranking for commercial-intent queries around AI voice agents, agentic workflows, and local SEO for service businesses.

## Locked positioning

These were decided before this brainstorm and are not revisited:

- **Target audience:** broad SMBs across Canada and the US (not Vancouver-only, not vertical-locked)
- **Content volume:** ~2 posts/month, with optionality to scale to programmatic in a future phase
- **Voice:** founder/solo-operator brand, named bylines on every post
- **First-3-month lead cluster:** pest-control vertical depth → AI voice agent horizontal → operator POV (in that order)
- **Client naming convention:** anonymized but specific. Lead client referenced as "a Metro Vancouver pest control operator." Prior agency referenced as "a national pay-per-lead agency charging $700 setup + $219/mo."
- **Conversion model:** discovery call + newsletter signup only. No lead magnets, no gated content in phase 1.
- **Build approach:** multi-page Astro with markdown Content Collections. No CMS.

## Tagline (canonical)

> AI voice agents, agentic workflows, and local SEO for service businesses.

Used in: hero subtitle, OG meta description, schema `slogan`, footer.

## Section 2 — Technical SEO foundation

### Per-page meta

Every page emits, via the Astro layout:

- `<title>` — page-specific, ≤ 60 chars, follows the pattern `{Page Title} — Clear Slope Digital`
- `<meta name="description">` — page-specific, 140–160 chars
- `<link rel="canonical">` — absolute URL of the current page
- Open Graph: `og:title`, `og:description`, `og:image`, `og:url`, `og:type` (`website` for marketing pages, `article` for `/notes/*` and the case study)
- Twitter card: `twitter:card="summary_large_image"`, `twitter:title`, `twitter:description`, `twitter:image`

### Schema (JSON-LD, embedded inline per page)

| Page | Schema types |
|------|--------------|
| `/` | `Organization` + `LocalBusiness` (using verified GBP data) + `WebSite` + `BreadcrumbList` |
| `/services/*` | `Service` + `FAQPage` + `BreadcrumbList` |
| `/work/pest-control-vancouver` | `Article` + `Person` (author) + `BreadcrumbList` |
| `/about/` | `Person` + `BreadcrumbList` |
| `/notes/[slug]` | `Article` + `Person` (author) + `BreadcrumbList` |
| `/notes/` (index) | `CollectionPage` + `BreadcrumbList` |

Schema nodes are linked via `@id`: `Organization.founder → Person`, `Person.worksFor → Organization`, `Article.author → Person`, `Article.publisher → Organization`. This is what builds the entity graph Google reads.

### Crawl & indexability

- `sitemap.xml` — auto-generated by `@astrojs/sitemap`
- `robots.txt` — allow all, link to sitemap
- `/notes/rss.xml` — RSS feed for the notes collection

## Section 3 — Content plan

### 3A. Service pages

Each follows a shared template, rendered from `src/content/services/*.md`:

| Page | H1 | Word target |
|------|----|----|
| `/services/ai-voice-agent` | "AI Voice Agents for Service Businesses" | 1,200–1,800 |
| `/services/agentic-workflows` | "Agentic Workflows: Custom AI That Runs Your Back Office" | 1,200–1,800 |
| `/services/local-seo` | "Local SEO for Service Businesses" | 1,500–2,200 |

**Shared structure (all three):**

1. Above-the-fold: H1 + 2-sentence summary + primary CTA (book call) + secondary anchor link to FAQ
2. Problem section (industry-specific pain, with proof point if available — e.g., All Greens 1,346 classified calls for `ai-voice-agent`)
3. How it works (mechanism, not product features)
4. Where it fits (industries / company size signals)
5. FAQ (5–8 questions, schema-marked)
6. Closing CTA (book call)

**Common rules:**
- Schema: `Service` + `FAQPage` + `BreadcrumbList`
- Internal links: outbound to relevant `/notes/` posts in the same cluster, plus the case study page
- No pricing table on-page (kept as discovery-call topic)
- No vendor logos / "as seen on" placeholder content

### 3B. Case study page (`/work/pest-control-vancouver`)

The anchor proof point. Drawn from the All Greens dataset, anonymized per the locked naming convention.

- **H1:** "How a Metro Vancouver pest control operator replaced a $700-setup, $219/month national pay-per-lead service with custom AI infrastructure"
- **Word target:** 2,500–3,500
- **Structure:** Context (operator profile, market, prior agency situation) → Problem (call volume vs. revenue conversion, the 1,346 classified calls reality) → What we built (voice agent + workflow + local SEO stack) → Results (call-classification numbers, leads, time saved — only metrics defensible from source data) → What we'd do differently → FAQ
- **Schema:** `Article` + `Person` (author) + `BreadcrumbList`
- **Anchoring assets:** Anonymized SearchKings dashboard screenshots (cropped/blurred to remove identifying info), call-classification chart, before/after timeline
- **Source data:** `/home/user/Projects/Clients/All Greens Pest Control/Analysis/summary.txt` and prior-agency reference materials in the same directory

**Constraint:** No claim is made on the page that cannot be backed by the source data.

### 3C. First 6 blog posts (months 1–3)

Cluster sequence: **A (pest-control vertical depth) → B (AI voice agent horizontal) → D (operator POV)**.

**Month 1 — Cluster A (pest-control vertical depth):**

1. `how-pest-control-operators-lose-revenue-to-missed-calls` — data-driven post anchored to the 1,346 classified calls. 1,500–2,000 words. Internal links: case study, `/services/ai-voice-agent`.
2. `pest-control-seo-the-phonebook-effect-is-not-enough` — why GBP/Maps is a floor, not a ceiling. 1,800–2,400 words. Internal links: `/services/local-seo`, case study, post #1.

**Month 2 — Cluster B (AI voice agent horizontal):**

3. `when-an-ai-voice-agent-actually-pays-for-itself` — unit-economics breakdown. 1,400–1,800 words. Internal links: `/services/ai-voice-agent`, case study.
4. `what-an-ai-voice-agent-cannot-do-and-shouldnt-pretend-to` — honest limitations, trust-building counter-positioning. 1,200–1,600 words. Internal links: `/services/ai-voice-agent`, post #3.

**Month 3 — Cluster D (operator POV):**

5. `why-i-charge-fixed-fees-not-pay-per-lead` — operator-POV manifesto, anonymized SearchKings comparison. 1,500–2,000 words. Internal links: case study, `/services/local-seo`, `/about/`.
6. `the-three-questions-i-ask-before-taking-on-a-service-business` — qualifying filter, founder voice. 1,200–1,500 words. Internal links: `/about/`, all three `/services/*`.

### 3D. Cadence + drafting workflow

- **Frequency:** 2 posts/month, published Tuesday morning (lock Tuesday for editorial consistency)
- **Drafting:** Mo drafts outline → AI assist drafts full post → Mo edits voice → publish
- **Storage:** `src/content/notes/{slug}.md` per Astro Content Collections
- **Frontmatter:** `title`, `description`, `slug`, `publishDate`, `updatedDate`, `author`, `cluster` (`pest-control` | `voice-agent` | `operator-pov`), `readingTime`, `featured` (bool)
- **Quality bar before publish:** original data or original POV (not regurgitation), at least 2 internal links, schema valid, no AI tells in voice
- **Review checkpoint:** after post 6, audit which cluster pulled best (impressions, position, conversion) and re-weight months 4–6 accordingly. Months 4+ are deferred — this spec does not lock them.

### 4B. About page (`/about/`)

Single-column, ~600–900 words. Structure:

1. **H1:** "About Mo Habib"
2. **Lede (2 sentences):** who Mo is, what Clear Slope does, who it's for. Pull directly from the locked tagline.
3. **Background (one short section):** mechanical engineering → applied AI / SEO. Frame the ME background honestly as **systems thinking applied to a new domain**. Reinforces the brand ("we build the systems your business has been trying to hire for"). No fake credentials.
4. **What I work on:** the three services as plain English, each one sentence.
5. **How I work:** founder-direct, fixed-fee, named clients become anonymized in writing, small client roster.
6. **Where to find me:** LinkedIn, contact form, calendar link.
7. **Inline schema block:** the `Person` JSON-LD above.

This page is the canonical destination for `Person.url`. Every byline links here. Every footer "About" link goes here.

## Section 5 — Conversion infrastructure

### 5A. Discovery call (primary CTA)

- **Tool:** Cal.com (free tier, inline embed)
- **Page:** `/book/` — full-page Cal.com embed, plus the embed referenced via component on the home page hero, the bottom of each service page, the bottom of the case study, and in the `<AuthorBox />`
- **Event type:** 30-minute discovery call (to be created in implementation)
- **Setup task** (deferred to implementation phase): create Cal.com account, create event type, capture event URL for the spec

### 5B. Newsletter signup (secondary CTA, deferred infrastructure)

- **Provider (when activated):** Buttondown
- **Phase 1 build:** newsletter form ships visually and submits to the **same Web3Forms endpoint already wired up in `ContactForm.astro`**, with subject line *"Newsletter signup — clearslopedigital.com"*. Mo receives one email per signup, manually imports into Buttondown when activated.
- **Component contract:** `<NewsletterForm />` exposes a single `email` input + submit button. The form's `action` URL is the only thing that changes when migrating to Buttondown's API — the migration is a 5-line change.

### 5C. CTA placement rules (Restrained — option A)

- **Per-page:** one primary CTA (book call) above the fold, one repeated CTA at the end. No middle-of-page CTAs on service pages.
- **Newsletter form appears on:**
  - `/notes/` index (above the post list)
  - End of each `/notes/[slug]` (after the `<AuthorBox />`)
  - Site footer
- **Site-wide:** footer is the only site-wide CTA surface
- **Explicitly excluded:** sticky bars (mobile or desktop), exit-intent popups, scroll-triggered modals, mid-post inline newsletter forms

This restraint is a hard constraint, not a preference — the operator-POV brand (especially posts 5 and 6) is incompatible with growth-hacker conversion patterns.

## Out of scope (phase 1 non-goals)

Explicit list of things this spec does **not** cover, to prevent scope creep during implementation:

- Per-location landing pages (deferred to phase C, post-traffic-validation)
- Programmatic content
- Lead magnets, gated content, or downloadable assets
- A/B testing infrastructure
- Heatmap / session replay (PostHog, Hotjar, etc.)
- Live chat
- Pricing pages or pricing tables
- Client testimonials section (until enough anonymized clients exist to populate one credibly)
- Multi-language / localization
- A blog comments system
- Author roster beyond Mo (single-author site in phase 1)
- Email marketing automation flows (drip sequences, etc.)
